# 容錯自動移轉 — 單機模擬 demo

在**一台**機器上用容器模擬 3 節點 quorum,展示 B4A 容錯自動移轉**機制的正確性**——
完全本地、免費、可重現,不需要任何付費 VPS、不碰 production。

## 跑法

```bash
bash demo.sh          # 跑完整 demo(選主 → 殺主接手 → 分區自我 fence)
bash demo.sh down     # 拆除
```
需求:一台有 Docker + docker compose 的機器(筆電即可)。

## 它展示什麼(對應真實架構)

| demo 元件 | 對應真實 B4A |
|---|---|
| 3 個 etcd(quorum) | 筆電 + 2 VPS 的共識/見證層(第三票破解 2 節點腦裂) |
| node-a / node-b | 2 個 broker 實例(主 / 熱備) |
| lease(TTL 5s) | 「誰是主」的租約;主死後租約過期 ⇒ 備機接手(RTO ≈ TTL) |
| lease-guarded 寫 `/b4a/work` | **lease-gated 真錢副作用**:下單前先確認「我還是主」,否則停手(自我 fence) |
| `/b4a/work` 單調遞增 | 狀態交接 + **單一寫者證明**:同一時刻只有一個主在寫、副作用不會被兩個主重複執行 |

## 三個驗證點

1. **選主**:啟動後同時只有一個 PRIMARY(etcd lease 互斥)。
2. **殺主自動接手**:`docker kill` 主節點 → lease 過期 → 備機自動升為主、`work` 繼續遞增(RTO ≈ lease TTL)。舊主拉回來不搶主(避免雙主)。
3. **分區自我 fence(防腦裂)**:把現任主從網路分區 → 它連不到 quorum、無法續租 → **自我 fence、停止寫入**(即使程序還活著)→ 對側接手。`work` 全程沒有被兩個節點重複寫 ⇒ **無腦裂、無雙下單**。

## 為什麼這是有效的證明(而非玩具)

核心難題是「**保證同一時刻只有一個主在發真錢副作用**」。這個 demo 證明的正是這條:
- 用**真的共識store(etcd)**做選主,不是手刻(手刻 fencing 最易出致命錯)。
- **自我 fence 用 lease-guarded 寫**繞過「第三方下游(交易所/Discord)無法 token-fence」的死結——主自己在失去租約時停手,不需要下游配合。
- 單機跑只能證明**機制正確性**,不能證明「實體機死掉也活」(那要真的多機);但對「驗證設計可運作」的報告目的,這完全足夠且零成本。

真實部署只是把這 5 個容器**散到 3 台機器**(筆電 + 2 VPS),機制完全一樣。
