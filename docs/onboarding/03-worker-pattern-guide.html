1. Worker 在三層架構的位置

Benson canonical 三層

High-level coordinator（LLM 規劃）→ Broker（control plane、capability registry、approve gate）→ Worker（執行 capability、實際做事）

Worker 是執行層—— broker 把任務派下來、worker 跑、回結果。 broker 不直接連外部 API、不算 K 線、不下單，全部透過 worker。這個分離讓：

  • 外部 API key 只在 worker container 裡（broker 不需要交易所密碼）
  • worker 可以單獨升級 / scale / 重啟、不影響 broker
  • broker 可以同時掛多種 worker（quote / strategy / risk / trading…）
  • worker crash 不會炸 broker、broker 看 worker disconnect 自動切備援
判斷你該不該寫 worker 而不是 broker endpoint 需要呼叫外部 API、跑 CPU 重計算、跑長時間 job、或要連 GPU / 特殊硬體 → 寫 worker。 純查 broker DB / 算 metric / 回 dashboard 用 → 寫 broker endpoint。

2. worker-sdk 的兩個核心元件

看 packages/csharp/worker-sdk/，整個 SDK 就兩個檔：

檔案角色
WorkerHost.cs 負責跟 broker 建 TCP 長連線、heartbeat、收任務、派給 handler、回結果。你不用碰它。
ICapabilityHandler.cs 你實作的介面、每個 capability 一個 handler。

ICapabilityHandler 介面（就 2 個成員）

public interface ICapabilityHandler
{
    string CapabilityId { get; }

    Task<(bool Success, string? ResultPayload, string? Error)> ExecuteAsync(
        string requestId, string route, string payload, string scope, CancellationToken ct);
}
  • CapabilityId：broker 用這個名字 lookup 你（例："quote.prices"、"trading.account"）
  • route：同一 capability 內的子操作（例："read_file" vs "list_dir"）
  • payload：broker 傳來的 JSON 字串、你自己 parse
  • scope：操作範圍 JSON（user_id / channel 等 metadata、目前多數 worker 沒用）
  • 回傳：(true, jsonResult, null) 成功；(false, null, "錯誤描述") 失敗
命名慣例：CapabilityId 用 domain.action 形式、小寫 + 點分隔。例： quote.prices、strategy.signal、trading.account、file.read。 這個 ID 全 broker 唯一、之後 high-level coordinator 也是用它呼叫你。

3. 寫第一個 Handler

看實際的範例：packages/csharp/workers/quote-worker/Handlers/QuotePricesHandler.cs

using System.Text.Json;
using WorkerSdk;
using QuoteWorker.Queue;

namespace QuoteWorker.Handlers;

public class QuotePricesHandler : ICapabilityHandler
{
    private readonly QuoteJobQueue _queue;
    public string CapabilityId => "quote.prices";

    public QuotePricesHandler(QuoteJobQueue queue) => _queue = queue;

    public Task<(bool Success, string? ResultPayload, string? Error)> ExecuteAsync(
        string requestId, string route, string payload, string scope, CancellationToken ct)
    {
        var prices = _queue.GetLatestPrices();
        var json = JsonSerializer.Serialize(new { count = prices.Count, quotes = prices });
        return Task.FromResult<(bool, string?, string?)>((true, json, null));
    }
}
Handler 設計三原則 1. Handler 只負責「翻譯 payload → 呼叫業務邏輯 → 序列化結果」、業務邏輯放別的 class。
2. 回 JSON、不回 C# object。broker 那端是 string-based protocol。
3. requestId 拿來寫 log 就好（要 audit 是 broker 的事、不是 worker）。
Handler 裡直接 new HttpClient()、寫死 API URL、不接 CancellationToken。 結果：connection leak、無法測、shutdown 卡住。正確做法：把外部依賴（HttpClient、DB、Queue）用建構子注入、handler 純做翻譯。

4. 組裝 Worker Program.cs（5 步驟）

看 packages/csharp/workers/quote-worker/Program.cs，骨架是這 5 步：

Step 1 — 讀設定
var config = new ConfigurationBuilder()
    .SetBasePath(Directory.GetCurrentDirectory())
    .AddJsonFile("appsettings.json", optional: true)
    .AddEnvironmentVariables("WORKER_")   // 例如 WORKER_BrokerHost=broker
    .AddCommandLine(args)
    .Build();
Step 2 — Logger
using var loggerFactory = LoggerFactory.Create(b =>
{
    b.AddConsole();
    b.SetMinimumLevel(LogLevel.Information);
});
var logger = loggerFactory.CreateLogger<WorkerHost>();
Step 3 — WorkerHostOptions
var options = new WorkerHostOptions
{
    BrokerHost               = config.GetValue<string>("Worker:BrokerHost") ?? "broker",
    BrokerPort               = config.GetValue("Worker:BrokerPort", 7000),
    WorkerId                 = $"my-wkr-{Guid.NewGuid():N}"[..20],
    MaxConcurrent            = 4,
    HeartbeatIntervalSeconds = 5,
    WorkerType               = "my-worker",
    WorkerAuthKeyId          = config.GetValue<string>("Worker:Auth:KeyId") ?? "",
    WorkerAuthSharedSecret   = config.GetValue<string>("Worker:Auth:SharedSecret") ?? "",
};
WorkerAuth 必填：broker 走 HMAC 驗證 worker、KeyId + SharedSecret 兩個必須跟 broker 那邊的 Worker:Auth:Workers 設定對得起來。沒設 worker 連得上 broker 但 broker 會直接 reject。
Step 4 — 建 host、註冊 handlers
var host = new WorkerHost(options, logger);
host.RegisterHandler(new MyCapabilityHandler(dep1, dep2));
host.RegisterHandler(new MyOtherHandler(dep3));
// 你一個 worker 可以掛多個 capability、有共用 dep 才一起放
Step 5 — 跑
using var cts = new CancellationTokenSource();
Console.CancelKeyPress += (_, e) => { e.Cancel = true; cts.Cancel(); };

await host.RunAsync(cts.Token);
// RunAsync 會 block 到 ct 取消為止、TCP 斷線自動重連

完整的 Program.cs 約 100 行（含註解）、看 quote-worker 那份照抄就行。

5. Containerfile 與 compose 註冊

Worker 都是獨立 container。看 packages/csharp/workers/quote-worker/Containerfile 抄一份， 改 csproj 路徑跟 entry：

FROM mcr.microsoft.com/dotnet/sdk:8.0 AS build
WORKDIR /src
COPY packages/csharp/ .
RUN dotnet publish workers/my-worker/MyWorker.csproj -c Release -o /app

FROM mcr.microsoft.com/dotnet/runtime:8.0
WORKDIR /app
COPY --from=build /app .
ENTRYPOINT ["dotnet", "MyWorker.dll"]

掛進 compose

編輯 deploy/compose.yml（或對應 stack），加一個 service：

my-worker:
  build:
    context: ../..
    dockerfile: packages/csharp/workers/my-worker/Containerfile
  environment:
    WORKER_BrokerHost: broker
    WORKER_BrokerPort: 7000
    WORKER_Worker__Auth__KeyId:        "my-wkr-key-01"
    WORKER_Worker__Auth__SharedSecret: "${MY_WORKER_SECRET}"   # 從 .env 帶
  depends_on:
    - broker
  restart: unless-stopped

broker 那邊註冊允許的 KeyId / Secret

編輯 packages/csharp/broker/appsettings.json（或環境變數）：

"Worker": {
  "Auth": {
    "Workers": [
      { "KeyId": "my-wkr-key-01", "SharedSecret": "${MY_WORKER_SECRET}" }
    ]
  }
}
把 SharedSecret 硬寫在 appsettings.json 推上 git。正確做法：寫 ${MY_WORKER_SECRET} 占位符、實際值放 .env 或 docker secret。

6. 驗證：broker 看得到嗎？

  1. 看 worker log：docker logs -f my-worker，應該看到「Connected to broker」+ 定時 heartbeat
  2. 看 broker log：docker logs -f broker | grep my-wkr，應該看到 register 成功
  3. 查 broker API：curl http://localhost:5100/api/v1/workers，列表應該有你
  4. 查 capability registry：curl http://localhost:5100/api/v1/capabilities，你的 CapabilityId 應該在裡面
  5. 實際 dispatch 試一次：curl -X POST http://localhost:5100/api/v1/workers/my-worker/my.capability -d '{"key":"value"}'
跑不起來的 debug 順序 1. Worker log 有沒有「Connected」？沒 → 網路 / port / WorkerAuth 三個檢查。
2. Broker log 有沒有「Worker registered」？沒 → KeyId/SharedSecret 對不上。
3. /api/v1/capabilities 有沒有你？沒 → handler 沒 RegisterHandler。
4. Dispatch 200 但結果空 → handler ExecuteAsync 邏輯有 bug、看 worker log。

7. 3 個常見踩雷

① BrokerHost 寫成 localhost

本機開發 OK、進 compose 就掛——container 裡的 localhost 是自己不是 broker。寫 service name broker、靠 compose internal DNS 解析。

② 多副本 worker 但 WorkerId 寫死

broker 用 WorkerId 區分連線、寫死的話第二副本上線會踢掉第一副本。對的做法：用 $"my-wkr-{Guid.NewGuid():N}"[..20] 每次啟動隨機生。

③ Handler 不接 CancellationToken

shutdown 時 broker 會傳 cancel，handler 沒 honor 的話跑滿 timeout 才被砍、整個 graceful shutdown 卡 30 秒。 長 task / HTTP call 都記得 pass ct：

// 對
await _httpClient.GetAsync(url, ct);
// 錯
await _httpClient.GetAsync(url);

8. 推薦讀的 5 個 Benson canonical 檔案
