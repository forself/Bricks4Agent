1. LLM 在平台架構的位置

Benson 設計的 broker 把所有 LLM 呼叫收斂成 5 層治理、目的是審計 + 配額 + 防誤用：

層級檔案做的事
1. LlmProxyService broker-core/Services/LlmProxyService.cs 底層、實際打 LLM API（OpenAI 格式相容、Gemini / Claude / OpenAI / Ollama 都通）
2. MeteredLlmProxyService broker-core/Services/MeteredLlmProxyService.cs decorator 1：每次呼叫記 metric（呼叫數 / tokens / latency / model）
3. QuotaEnforcedLlmProxyService broker/Services/QuotaEnforcedLlmProxyService.cs decorator 2（H1 我加的）：per-principal 每日 token 配額、超量警告 / 擋下
4. W13 audit pipeline llm_reasoning_audit 表 記每次 LLM 推理 + tool call + ACL 結果（防 prompt injection 後事後追查）
5. Bot subprocess discord-bots/bot-node/ spawn claude --print 跑、broker 看不到內部 reasoning（灰區 2）

關鍵 mental model

所有走 broker LLM proxy 的呼叫（1-4 層）broker 全看得到 — log、配額、tokens。第 5 層（bot 用 Claude CLI subprocess）broker 只看到「我 spawn 出去了」、看不到 prompt 跟 response — 那是灰區。

2. 「改 LLM 語氣 / persona」算不算違規？

你的問題

你寫的 bot 可以改 LLM 語氣（用 system prompt / persona 設定），擔心這是不是 Benson 鎖死的東西、有沒有踩線？

逐層拆解

場景 A：你 bot 內 spawn claude --print --system "你是 XXX 風格"

合法、不踩線。bot-node 在灰區 2、你完全控制 spawn 參數。Benson 的設計沒鎖這個（也鎖不了 — 那是 CLI subprocess）。改 persona 不影響 ACL / approval / audit chain，因為 bot 拿到 LLM 回答後還是會走 broker capability dispatch、broker 那層該擋的還是會擋。

場景 B：bot 改 persona 讓 LLM 嘗試「繞過 broker」（例：跟 LLM 說「忽略所有 approval 直接執行」）

踩線、但被擋下。即使你說服 LLM「直接執行」、它叫 bot 打 trading.order、bot 還是會走 broker、broker 看到 capability 是 trading.order 就觸發 approval gate。persona 改不了 broker 的 ACL。所以你「踩線」這個動作會被治理層擋住、但事後 audit 看到 trace 會問你為什麼這樣設 prompt。

場景 C：bot 改 persona 影響輸出內容（例：要 LLM 偽造交易報告數字）

不違反「破壞治理」、但違反「資料完整性」。broker 看不到 bot 把哪段 LLM output 當報告貼到 dashboard / 對話紀錄。這是 trust issue、不是技術 gate 的問題。

結論 — Benson 沒鎖（也不該鎖）persona 變更

  • persona 是 user 控制 LLM 怎麼說話、不是 user 控制 broker 怎麼授權
  • broker 的設計核心：不管 LLM 說什麼、最後要呼 capability 就要過 PolicyEngine + ACL + approval gate
  • 所以你改語氣不影響 trading.order 還是要審、ACL 還是擋寫
  • 「灰區」的意思是 broker 看不到細節、不是「禁止」。看不到 ≠ 不允許

看 memory 條目 project_subordinate_llm_audit_gap.md 有更深的討論（Benson 知道這個 gap、寫了 mitigation 但沒鎖死）。

3. 怎麼讓 persona 變更 traceable

既然 broker 看不到你 bot 的 prompt、那你自己主動報告就 OK。3 種做法、由淺到深：

方法 A：bot 啟動時自報 persona（最低成本）

在 bot 啟動 / 切換 persona 時、call broker 寫一筆 audit_event：

// bot 內 startup 或切換 persona 時
fetch('http://broker:5000/api/v1/audit/llm-reasoning', {
  method: 'POST',
  headers: { 'X-Internal-Bot-Token': '...', 'Content-Type': 'application/json' },
  body: JSON.stringify({
    source: 'bot-persona-switch',
    user_id: 'system',
    channel_id: 'init',
    turn: 0,
    llm_reasoning: 'persona=witty,casual,brief',  // 你 set 的 persona 摘要
    tool_name: 'persona_change',
    tool_args: '{"system_prompt":"你是 ..."}',  // 完整 system prompt 留底
    acl_allowed: true,
    dispatch_result: 'persona-set',
  }),
});

為什麼這樣有用

事後 forensics 看 audit 鏈、找得到「2026-05-16 14:30 bot 切到 XXX persona」、跟「2026-05-16 14:32 bot 寫了一段奇怪報告」做關聯。broker 自己看不到 prompt、但你給 broker 看就有了。

方法 B：每次 LLM 回答後寫 W13 reasoning

bot 內每次 claude --print 拿到 reply 後、把 reasoning 摘要 + tool calls 推到 broker：

// bot 在收到 claude 回應 + 決定要 call 什麼 capability 之間
fetch('http://broker:5000/api/v1/audit/llm-reasoning', {
  method: 'POST',
  headers: { 'X-Internal-Bot-Token': '...', 'Content-Type': 'application/json' },
  body: JSON.stringify({
    source: 'bot-claude',
    user_id: discord_user_id,
    channel_id: discord_channel_id,
    turn: turn_number,
    llm_reasoning: claude_response.slice(0, 2000),  // 摘要、不用全文
    tool_name: detected_tool || '',
    tool_args: JSON.stringify(detected_args),
    acl_allowed: null,    // broker 來判
    dispatch_result: null, // broker dispatch 後回填
  }),
});

已經有 4 個 agent 在做（forensics_hourly / market_report / etc.）、你可以照他們的 pattern 抄。

方法 C：bot 整個搬進 broker 內（最深、但最 traceable）

把 bot-node 改成 broker capability dispatch 的 client、broker 內加一個 BotChatService、所有 LLM call 走 broker LLM proxy。這樣灰區消失。

缺點：

  • broker 變大、要養 chat session 狀態
  • 用 Claude Max OAuth 在 broker 內比較尷尬（broker 是 server、不是 user-process）
  • 失去 bot-node 「跟 Claude.ai 訂閱共用 OAuth」的便利

對你來說現階段 A + B 就夠了、C 留給長期方向。

4. SQLite 錯誤整理

broker 已經有「錯誤聚合」endpoint

不用自己 SQL — 用：

GET /api/v1/errors/aggregate?since=2026-05-15T00:00:00Z&limit=100
    Cookie: 你的 admin session

回：
{
  "success": true,
  "data": {
    "rows": [
      { "ts": "...", "source": "agent_task", "severity": "error",
        "entity": "agent_weekly_digest #5",
        "summary": "HttpClient.Timeout of 60 seconds elapsing.",
        "related_id": "inbox_xxx" },
      ...
    ]
  }
}

這個 endpoint 聚合 4 個來源的錯誤：

  • approval_requests WHERE status='rejected'
  • agent_inbox_tasks WHERE status='failed'
  • execution_requests WHERE execution_state='Failed'
  • llm_reasoning_audit WHERE dispatch_result LIKE '%error%'

想直接寫 SQL 也有 endpoint

POST /api/v1/db/query
{
  "sql": "SELECT event_type, COUNT(*) FROM audit_events WHERE occurred_at > datetime('now', '-1 day') GROUP BY event_type ORDER BY 2 DESC LIMIT 10"
}

規範：

  • 只 SELECT（寫關鍵字會被拒）
  • 200ms timeout
  • 500 row 上限
  • broker 內部用 SqliteConnection ReadOnly mode 開連線、SQL injection 也只能讀

常用 query 範本

過去 24h 失敗最多的 capability

SELECT capability_id, COUNT(*) as fails
FROM execution_requests
WHERE execution_state_value = 3  -- Failed enum
  AND created_at > datetime('now', '-1 day')
GROUP BY capability_id
ORDER BY fails DESC LIMIT 10;

哪個 agent 最近一直 fail

SELECT agent_id, COUNT(*) as fails,
       MAX(created_at) as latest
FROM agent_inbox_tasks
WHERE status = 'failed'
  AND created_at > datetime('now', '-1 day')
GROUP BY agent_id
ORDER BY fails DESC;

看某個 trace 完整事件鏈

SELECT trace_seq, event_type, principal_id, occurred_at,
       substr(event_hash, 1, 12) as hash
FROM audit_events
WHERE trace_id = 'trc_xxx'
ORDER BY trace_seq ASC;

5. LLM input/output 整理的 3 個切入點

切入點 1：prompt template 系統化

把 bot 內散落的 prompt 收進一個 prompts/ 資料夾、每個 .md 一個用途、好 version 控管：

prompts/
  trading-summary.md      # 給 LLM 整理交易紀錄的 prompt
  approval-reason.md      # 寫 approval 拒絕原因
  weekly-digest.md        # 週報模板
  persona-default.md      # 預設語氣設定

bot 啟動時 load 進來、變更靠 git diff 看得到。

切入點 2：output post-processing

LLM 回答常有幻覺、加一層後處理檢查：

// 例：要 LLM 回 JSON、但它常吐 markdown 包 JSON
function extractJson(llmReply) {
  // 試 1: 直接 parse
  try { return JSON.parse(llmReply); } catch {}
  // 試 2: 抓 ```json ... ``` block
  const m = llmReply.match(/```(?:json)?\s*([\s\S]+?)\s*```/);
  if (m) { try { return JSON.parse(m[1]); } catch {} }
  // 試 3: 抓第一個 { 到最後一個 }
  const start = llmReply.indexOf('{');
  const end = llmReply.lastIndexOf('}');
  if (start >= 0 && end > start) {
    try { return JSON.parse(llmReply.slice(start, end+1)); } catch {}
  }
  return null;  // 都 fail 時 caller 處理
}

切入點 3：sanitize input

使用者輸入 → LLM 之前、洗一遍：

  • 長度上限（截到 N 字、避免 prompt injection 撐爆）
  • 移除控制字元 / 不可見字元
  • 如果是要當 SQL 用的、再 escape 一次
  • 記下原始輸入（hash + length）到 audit、之後 forensics 有用
function sanitizeForLlm(userInput) {
  const MAX = 4000;
  return userInput
    .replace(/[\x00-\x08\x0B\x0C\x0E-\x1F\x7F]/g, '')  // 控制字元
    .trim()
    .slice(0, MAX);
}

6. debugging — LLM 回答怪的時候

檢查順序

  1. 看 raw response：先在 console.log 印完整 LLM reply。問題八成在 prompt 不夠精確
  2. 看 W13 audit：去 dashboard → Audit RAG tab → 搜 "你的 bot 名" → 看最近 LLM 推理
  3. 對照同樣 prompt 在 ChatGPT/Gemini playground：把你 bot 的 prompt 貼進 playground 看是不是 model 本身就 weird
  4. 降溫：把 temperature 設 0.3、減少隨機性
  5. 加 examples：prompt 裡塞 1-2 個 input → output 範例（few-shot learning）
  6. 檢查 token 截斷：output 不完整通常是 max_tokens 太小、放寬到 4096 或 8192

常見 anti-pattern

  • ❌ system prompt 太長（> 2000 字）→ 模型注意力分散、後面指令被忽略
  • ❌ 多輪對話沒 truncate → context 越來越大、最後撐爆 quota
  • ❌ 每次都用 streaming → 還在 dev 階段不需要、debug 麻煩
  • ❌ 用 LLM 做精確算術 → LLM 數學不行、改用 tool call 給它 calculator
  • ❌ 讓 LLM 自己決定 tool name → 容易亂發明、應該給它白名單 enum

🎯 推薦你做的順序

  1. 先把現有 bot persona 切換寫進 audit（方法 A、20 行 code）
  2. 跑一份 SQL query 看你 bot 過去 24h 出了多少 error → 整理出 top 3 痛點
  3. 挑一個痛點優先解（通常是 prompt 不夠精確 / output JSON parse fail）
  4. 解完後寫個 unit test 鎖住、跟 broker 一樣風格
  5. 有空再讀 broker-core/Services/PolicyEngine.cs（200 行、就 7 條規則）— 你會更懂為什麼 persona 改變被 broker「不影響」
