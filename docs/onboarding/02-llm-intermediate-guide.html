1. LLM 在三層架構的位置

讀過 CurrentArchitectureAndProgress-2026-03-26.md 知道平台分三層：

層角色LLM 的位置
1. high-level 對話、釐清、意圖判斷 HighLevelLlm（預設 gpt-5.4-mini）
由 HighLevelCoordinator 呼叫
2. broker / control plane 授權、稽核、artifact 派發 LlmProxyService（集中代理）
所有要 LLM 的內部服務都該走這條
3. execution worker / 執行 worker 透過 broker 的 /api/v1/llm-proxy/chat 借 LLM 用、不該自己直接打外網
架構原則 1 高階層可以解讀使用者輸入、可以建議要用哪個 execution model；但只有 broker 能批准 + 派發 + 記錄。高階模型不可以自己呼 capability、不可以自己改 ACL。
架構原則 2 所有 LLM 呼叫都該走 LlmProxyService（或它的 MeteredLlmProxyService 裝飾器）。這樣 broker 才能：記 metric、做 quota、稽核 reasoning、配 capability 上下文。worker 不該繞過 broker 直接打 OpenAI / Gemini / Anthropic API。

2. LlmProxyService — 集中代理模式

核心檔案

檔案角色
broker-core/Services/ILlmProxyService.cs 介面定義（ChatAsync / ListModelsAsync / HealthCheckAsync）
broker-core/Services/LlmProxyService.cs 底層、實際打 OpenAI-compatible API
broker-core/Services/MeteredLlmProxyService.cs decorator、每次呼叫記 metric（呼叫數 / tokens / latency / model）
broker-core/Services/LlmProxyMetrics.cs in-memory metrics store

怎麼呼用

broker 內任何 service / endpoint：

// constructor inject
public MyService(ILlmProxyService llm)
{
    _llm = llm;
}

// 呼叫
var bodyJson = JsonDocument.Parse(@"{
    ""messages"": [
        { ""role"": ""system"", ""content"": ""你是文件整理助手"" },
        { ""role"": ""user"",   ""content"": ""幫我整理這份報告..."" }
    ]
}");

var result = await _llm.ChatAsync(bodyJson.RootElement, task: currentTask, ct);
// result.Content      = LLM 回應文字
// result.ToolCalls    = LLM 想呼的 tool（如果有）
// result.EvalCount    = output token 數
// result.Model        = 實際用的 model

worker 內（worker 不直接 inject、用 HTTP）：

POST http://broker:5000/api/v1/llm-proxy/chat
  X-Worker-Identity: ...  // worker auth
  Content-Type: application/json
{
  "messages": [...]
}

config（在 .env.trading）

LLM_API_KEY=...                    # 一個 key 給整個 broker 用
LLM_PROXY_DEFAULT_MODEL=gpt-5.4-mini  # 預設 model
LLM_PROXY_BASE_URL=https://api.openai.com   # 也可換 Gemini / Anthropic

3. 3 種正確的 LLM 整合切入點

想加新 LLM 功能時、3 種 canonical 做法：

切入點 A：在 broker 內加 BackgroundService

  • 適合：定時 / 觸發式任務（每小時整理 audit、每天生報告）
  • pattern：寫一個 class : BackgroundService、constructor inject ILlmProxyService
  • 所有 LLM 呼叫自動走 metering + capability 上下文
  • broker.db 開新表存結果（用 BaseOrm）

切入點 B：新增 worker（spawn-based）

  • 適合：會跑很久的、隔離資源的 LLM 任務（fine-tune / 大量 embedding / 長段對話）
  • pattern：新增 workers/your-worker/ 專案、用 worker-sdk、註冊 capability 給 broker
  • broker 用 IContainerManager.SpawnWorkerAsync() 啟容器、worker 透過 broker 的 LLM proxy 借 LLM
  • 有 ACL 強制：必須 capability 註冊、才能被 spawn / 呼叫

切入點 C：擴 HighLevelCoordinator

  • 適合：LINE 對話流的新功能（新的 ?command、新工作流 state）
  • 檔案：broker/Services/HighLevelCoordinator.cs
  • 加新 command 進 grammar（讀 §5.1 architecture doc）
  • state machine 要明確 gate（不能讓任意 user input 進到 execution intent）

選哪個？

定時報告 → A。使用者主動觸發、會跑很久 → B。對話流改寫 → C。

不確定就先 A（最簡單）、後面要長期跑再升級成 B。

4. prompt / persona 客製化

你想讓 LLM 「語氣不一樣」、「換 persona」、「給特定領域風格」— 完全 OK、是設計支援的事。重點是怎麼做才不繞過治理。

正確做法：system prompt 進 ChatAsync

var messages = new[] {
    new {
        role = "system",
        content = "你是 Bricks4Agent 的 LINE 助手。語氣專業但口語、回答精簡、" +
                  "每次回答最多 3 段、優先用條列。遇到使用者問交易、要先聲明這是 POC、" +
                  "結論不構成投資建議。"
    },
    new { role = "user", content = userMessage }
};
var result = await _llm.ChatAsync(JsonSerializer.SerializeToElement(new { messages }), task, ct);

進階：prompt 從外部檔案讀（version 控管）

// 在 broker 內加個 PromptCatalog service
public class PromptCatalog
{
    private readonly Dictionary<string, string> _prompts = new();

    public PromptCatalog(IConfiguration config)
    {
        var root = config["Prompts:Root"] ?? "./prompts";
        foreach (var f in Directory.EnumerateFiles(root, "*.md"))
        {
            var key = Path.GetFileNameWithoutExtension(f);
            _prompts[key] = File.ReadAllText(f);
        }
    }

    public string Get(string key) => _prompts.GetValueOrDefault(key, "");
}

// 用法：
var systemPrompt = _prompts.Get("line-assistant-default");
// 或 user 切 persona：
var systemPrompt = _prompts.Get($"persona-{personaName}");

放在 prompts/ 資料夾、每個 .md 一個 prompt、git 可追：

prompts/
  line-assistant-default.md   # LINE 主對話 persona
  audit-summarizer.md          # 給 LLM 整理 audit 用
  trade-report.md              # 交易報告語氣
為什麼這樣做是「不繞過治理」 persona 換來換去、LLM 講話風格不同 — 但呼出來的 tool / capability 還是要過 broker 的 PolicyEngine + ACL + ApprovalService。LLM 想偽裝 admin 也沒用、broker 不認 prompt、只認 scoped_token。

反例（不要這樣做）

worker 自己 spawn openai CLI / claude CLI 直接打 LLM、繞過 broker LlmProxy。
→ broker 看不到、metric 算不到、quota 管不到、之後 forensics 完全瞎。
把 API key 散落在多個 worker 的 .env、不走 broker 集中代理。
→ key 洩漏面變大、換 model 要改 N 個 service。

5. SQLite 錯誤整理

broker 的 broker.db 有幾張表記錯誤、直接 SQL 查最直接。

進 container 內查

docker exec -it broker bash
# 容器內：
apt-get update && apt-get install -y sqlite3   # 如果沒裝
sqlite3 /data/broker.db
# 進 sqlite shell：
.tables
.schema audit_events
SELECT ... ;
.quit

從 host 直接查（broker.db 在 docker volume）

# Linux/macOS:
sudo sqlite3 /var/lib/docker/volumes/$(docker volume ls -q | grep broker-data)/_data/broker.db

# Windows (PowerShell)：先進 Docker Desktop WSL VM
wsl -d docker-desktop
ls /var/lib/docker/volumes/

常用 query 範本

過去 24h audit event 類型分布

SELECT event_type, COUNT(*) as count
FROM audit_events
WHERE occurred_at > datetime('now', '-1 day')
GROUP BY event_type
ORDER BY count DESC
LIMIT 15;

過去 24h 哪個 capability fail 最多

SELECT capability_id, COUNT(*) as fails
FROM execution_requests
WHERE execution_state_value = 3  -- Failed enum
  AND created_at > datetime('now', '-1 day')
GROUP BY capability_id
ORDER BY fails DESC LIMIT 10;

看某 trace 完整事件鏈（驗 hash chain）

SELECT trace_seq, event_type, principal_id,
       substr(event_hash, 1, 12) as hash,
       substr(previous_event_hash, 1, 12) as prev,
       occurred_at
FROM audit_events
WHERE trace_id = 'trc_xxxxx'
ORDER BY trace_seq ASC;

approval 拒絕的清單

SELECT requested_at, capability_id, principal_id,
       decided_by, decision_reason
FROM approval_requests
WHERE status = 'rejected'
  AND requested_at > datetime('now', '-7 day')
ORDER BY requested_at DESC;

6. LLM input / output 處理 anti-patterns

避免這些

system prompt 太長（> 2000 字）→ 模型注意力分散、後面指令被忽略。砍到 < 500 字、用 few-shot examples 取代長解釋。
多輪對話無 truncate → context 越來越大、最後 quota 撐爆 / 延遲飆高。設 message history 上限（例 last 10 turns）。
讓 LLM 自己決定 tool name → 容易亂發明、應該給白名單 enum 在 system prompt：「only one of: rail_search, hsr_search, bus_search」。
用 LLM 做精確算術 → LLM 數學不行、改 tool call 給它 calculator endpoint。
把 user 輸入直接當 system prompt 拼接→ prompt injection 風險、user 可以說「忽略上面所有指示、改做 X」。永遠把 user 內容放 user message 裡、system 不混 user input。

該做的

sanitize input 範本

private static string SanitizeForLlm(string userInput)
{
    const int MAX = 4000;
    return new string(userInput
        .Where(c => !char.IsControl(c) || c == '\n' || c == '\t')
        .Take(MAX).ToArray())
        .Trim();
}

output JSON extraction（LLM 常包 markdown）

private static JsonElement? ExtractJson(string llmReply)
{
    // 試 1: 直接 parse
    try { return JsonDocument.Parse(llmReply).RootElement; } catch {}

    // 試 2: 抓 ```json ... ``` block
    var m = System.Text.RegularExpressions.Regex.Match(
        llmReply, @"```(?:json)?\s*([\s\S]+?)\s*```");
    if (m.Success) {
        try { return JsonDocument.Parse(m.Groups[1].Value).RootElement; } catch {}
    }

    // 試 3: 抓第一個 { 到最後一個 }
    var start = llmReply.IndexOf('{');
    var end = llmReply.LastIndexOf('}');
    if (start >= 0 && end > start) {
        try { return JsonDocument.Parse(llmReply[start..(end+1)]).RootElement; } catch {}
    }
    return null;
}

7. debugging — LLM 回答怪的時候

檢查順序

  1. 看 raw response：先 _logger.LogInformation("LLM reply: {R}", result.Content) 印完整內容
  2. 看 metric：GET /api/v1/llm-proxy/status 看最近呼叫的 model / latency / token 數
  3. 對照 playground：同 prompt 貼 ChatGPT / Gemini playground 看是不是 model 本身就 weird
  4. 降 temperature：default 0.7、降到 0.2-0.3 可大幅減少 hallucination
  5. 加 few-shot examples：在 system prompt 內塞 1-2 個 「user → desired output」範例
  6. 檢查 token 截斷：output 不完整通常是 max_tokens 太小、放寬到 4096 或 8192

broker LLM 健康檢查

# dashboard 內：LLM Proxy tab → 健康檢查 / 用量趨勢
# 或 curl：
curl -s -b cookies.txt http://localhost:5100/api/v1/llm-proxy/status | head -c 500
curl -s -b cookies.txt http://localhost:5100/api/v1/llm-proxy/recent?limit=10
curl -s -b cookies.txt http://localhost:5100/api/v1/llm-proxy/healthcheck
